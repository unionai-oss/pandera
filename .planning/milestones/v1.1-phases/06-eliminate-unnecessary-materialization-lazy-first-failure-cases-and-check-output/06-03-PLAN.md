From pandera/backends/narwhals/base.py (failure_cases_metadata — current, lines 181–310):

The current method always converts to pl.DataFrame:
- Calls `_materialize(err.failure_cases)` → eager nw.DataFrame
- Calls `pl.from_arrow(fc_eager.to_arrow())` → pl.DataFrame
- Uses `pl.lit()`, `pl.Series()`, `pl.concat()` throughout

New failure_cases_metadata() design:

**Lazy/SQL path** (polars-lazy nw.LazyFrame, ibis nw.DataFrame with hasattr native "execute"):
- Row index is always None — no forced ordering
- Attach metadata columns using `nw.lit(value).alias(name)` via `.with_columns()`
- For single-column fc: use `.rename({old_name: "failure_case"})` then `.with_columns(nw.lit(...))`
- For multi-column fc: use `nw.concat_str()` to produce a human-readable string per row — no Arrow
  roundtrip needed; `nw.concat_str()` works cross-backend (polars and ibis) and stays fully lazy
- Final result: unwrap to native via `nw.to_native(enriched)` for the FailureCaseMetadata

**Eager path** (polars-eager nw.DataFrame — not LazyFrame, not SQL-lazy):
- Row index is derivable — compute from check_output using `.with_row_index()`
- Keep existing polars-based logic (pl.lit, pl.Series, pl.concat) — it works correctly for eager inputs
- This is acceptable at the boundary since check_output is already eager for this path

**Scalar failure_cases path** (not nw.LazyFrame or nw.DataFrame):
- Same as current: `pl.DataFrame(scalar_failure_cases)` is fine — these are scalar Python values

**Lazy/SQL detection** (from RESEARCH.md and Phase 01 STATE.md decisions):
```python
def _is_lazy_or_sql(fc) -> bool:
    """True for polars-lazy (nw.LazyFrame) or SQL-lazy (nw.DataFrame wrapping ibis.Table)."""
    if isinstance(fc, nw.LazyFrame):
        return True
    if isinstance(fc, nw.DataFrame):
        native = nw.to_native(fc)
        return hasattr(native, "execute")  # ibis.Table has .execute(); polars DataFrame does not
    return False
```

**Lazy path enrichment** (narwhals ops only):
```python
fc = err.failure_cases
col_names = fc.collect_schema().names()

if len(col_names) == 1:
    # Single-column: rename directly
    enriched = fc.rename({col_names[0]: "failure_case"})
else:
    # Multi-column: build a readable "col=value, col=value" string per row.
    # nw.concat_str() is cross-backend (polars and ibis) and stays lazy.
    parts = [
        nw.lit(f"{c}=").cast(nw.String) + nw.col(c).cast(nw.String)
        for c in col_names
    ]
    enriched = fc.select(nw.concat_str(*parts, separator=", ").alias("failure_case"))

enriched = enriched.with_columns(
    nw.lit(err.schema.__class__.__name__).alias("schema_context"),
    nw.lit(err.schema.name).alias("column"),
    nw.lit(check_identifier).alias("check"),
    nw.lit(err.check_index).cast(nw.Int32).alias("check_number"),
    nw.lit(None).cast(nw.Int32).alias("index"),
)
failure_case_collection.append(nw.to_native(enriched))
```

Note: the multi-column format (`"a=1, b=2"`) is intentionally different from the polars eager path
(`'{"a":1,"b":2}'` JSON struct). This is acceptable — the polars-eager path retains its existing
behavior; the lazy/SQL path produces a readable plain-string format. Both are string representations
of the failing row. There is no shared contract requiring JSON.

**Concatenation** — errors from a single validate() call all come from the same backend, so
failure_case_collection will be homogeneous. Use the type of the first item to pick the concat:
```python
if failure_case_collection:
    first = failure_case_collection[0]
    if hasattr(first, "union"):  # ibis.Table
        import functools
        failure_cases = functools.reduce(lambda a, b: a.union(b), failure_case_collection)
    else:
        failure_cases = pl.concat(failure_case_collection)  # pl.LazyFrame or pl.DataFrame
```
