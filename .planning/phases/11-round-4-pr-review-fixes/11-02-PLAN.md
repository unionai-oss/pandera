<objective>
Update PySpark tests so they handle the new SchemaErrors-raising contract that Plan 01 introduced. Introduce a backend-aware helper `validate_collecting_errors(schema, df, **kwargs)` in `tests/pyspark/conftest.py` that returns an `(out_df, errors_dict)` tuple regardless of whether validation raised SchemaErrors (narwhals path) or attached errors to `df.pandera.errors` (native PySpark path). Replace every `.pandera.errors` assertion in tests/pyspark and the 4 sites in tests/narwhals/test_e2e.py with assertions against the helper's tuple return.

Purpose: PR #2339 round-4 review SE-02 — after Plan 01 removes the `is_pyspark` branch, PySpark Narwhals validation raises SchemaErrors. Without this plan, ~64 assertions across 7 PySpark test files break under PANDERA_USE_NARWHALS_BACKEND=True. The helper approach keeps the native backend test surface intact while making the narwhals path pass too.

**Cross-directory import note (accepted, NOT a blocker).** `tests/narwhals/test_e2e.py` will import `validate_collecting_errors` from `tests.pyspark.conftest`. This adds one cross-domain import. It is accepted because: (1) `test_e2e.py` already imports `pa_pyspark` (PySpark schema constructors) and creates PySpark fixtures inline, so it is already coupled to the PySpark test domain; (2) the helper is specifically a PySpark validation helper — it has no meaningful home outside `tests/pyspark/`; (3) moving it to a shared `tests/conftest.py` would force the PySpark/SchemaErrors imports into the global test fixture scope, which is worse coupling than the current targeted cross-directory import. If this becomes problematic in future, the migration path is to relocate the helper to a new `tests/_shared/pyspark_helpers.py` module and update the two callers; that is explicitly out of scope for Phase 11.

Output: Updated test files + new helper in conftest.py; both nox sessions (`tests(pyspark)` and `tests_narwhals_backend(pyspark)`) pass.
</objective>
