<objective>
Achieve behavioral parity between the narwhals PySpark validation path and the
native PySpark backend for two pre-merge defects identified in PR #2339 review:

1. CORR-01: `strict='filter'` must return the column-filtered DataFrame on the
   success path (currently returns the original unfiltered `check_obj`).
2. CORR-02: `df.pandera.schema` must be set on the returned PySpark DataFrame
   (currently `add_schema(schema)` is never called by the narwhals path).

Both defects share the same edit site — `_handle_pyspark_validation_result`
and its two call sites in `DataFrameSchemaBackend.validate()` — so they are
grouped into one plan per RESEARCH.md guidance.

Purpose: Land the narwhals PySpark success/error contract changes required
before PR #2339 can merge. Both fixes are root-cause: the column filter
already happens on `check_lf`, and `PanderaPySparkAccessor.add_schema` is
already idempotent — only the propagation back to native is missing.

Output:
- `pandera/backends/narwhals/container.py` with corrected PySpark return path.
- `tests/pyspark/test_pyspark_model.py` and `tests/pyspark/test_pyspark_accessor.py`
  with the temporary `@pytest.mark.xfail(condition=CONFIG.use_narwhals_backend, ...)`
  decorators removed for the two now-passing tests.
- `tests/narwhals/test_phase01_arch.py` extended to assert the new
  `add_schema(schema)` behavior on both ARCH-04 success and error path tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → DataFrameSchema.validate() | User-supplied PySpark DataFrame crosses into pandera; schema is user-controlled but validated against pandera's accessor protocol |
| narwhals container.py → PanderaPySparkAccessor | Backend mutates user's DataFrame via `df.pandera.add_schema(schema)` and `df.pandera.errors = ...`; relies on accessor contract |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | `_handle_pyspark_validation_result` mutating `check_obj.pandera` state | accept | `add_schema` and `errors=` mutations are the documented PySpark contract (see `pandera/accessors/pyspark_sql_accessor.py`); behavior matches native PySpark backend. No new attack surface — only the existing accessor protocol is exercised. |
| T-05-02 | Information Disclosure | Schema object leaked via `pandera.schema` accessor on returned DataFrame | accept | Schema metadata being attached to validated DataFrames is the documented public API of the PySpark accessor; matches native backend behavior. No PII or secrets exposed. |
| T-05-03 | Denial of Service | `_to_frame_kind_nw(check_lf, return_type)` on PySpark return path triggering a Spark action | mitigate | `_to_frame_kind_nw` for PySpark returns `nw.to_native(lf)` without materialization (PySpark has `.collect()` so the `caller_was_eager_polars` branch is False — verified in RESEARCH.md `_to_frame_kind_nw` Behavior section). The lazy → native unwrap is metadata-only; no Spark job triggered. |
| T-05-04 | Tampering | Test xfail removal masking true regression | mitigate | Task 1 lands the code fix BEFORE Task 3 removes the xfails (sequential within-plan ordering); Task 2 updates ARCH-04 unit tests to assert the new behavior at the method level (independent of integration test); Task 3 verify step runs the formerly-xfailed tests to confirm they pass, not xpass. |
