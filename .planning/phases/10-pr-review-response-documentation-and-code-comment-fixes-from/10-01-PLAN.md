<objective>
Expand the existing narwhals opt-in note in `docs/source/pyspark_sql.md` (currently at lines 29-47) so that users opting into the narwhals backend for PySpark are warned about three undocumented behavioral differences vs. the native PySpark backend:

1. M1 (DOC-M1): `coerce=True` is a silent no-op under the narwhals backend — no coercion happens and no error is raised. This is because the narwhals `ColumnBackend` (in `pandera/backends/narwhals/components.py`) has no `coerce_dtype` step (deliberate — matches the polars narwhals backend contract).
2. M2 (DOC-M2): Custom checks written against `PysparkDataframeColumnObject` (i.e. taking `pyspark_obj: PysparkDataframeColumnObject` as input) are incompatible with the narwhals backend. The narwhals backend passes `NarwhalsData(frame, key)` to check functions instead.
3. M3 (DOC-M3): Error reporting follows the PySpark accessor convention (`df.pandera.errors`) and the frame is returned, rather than raising `SchemaErrors` — even when `lazy=True`. For the polars/ibis narwhals backends, `lazy=True` raises `SchemaErrors`; for the PySpark narwhals backend it does not. This is intentional behavioural parity with the native PySpark backend.

Purpose: Round-3 PR review (PR review 4393093479) flagged three undocumented behavioral differences that users opting in via `PANDERA_USE_NARWHALS_BACKEND=True` need to know about before adopting the narwhals backend for PySpark. Without this documentation, users will silently lose coercion behavior, get confusing errors when migrating custom checks, and be surprised by the error-surfacing contract under `lazy=True`.

Output: Updated `docs/source/pyspark_sql.md` with three new limitation bullets inside the existing narwhals opt-in note, alongside the existing element-wise/sample/tail limitations bullet.
</objective>
